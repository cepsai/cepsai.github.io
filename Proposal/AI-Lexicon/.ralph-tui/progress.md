## Codebase Patterns (Study These First)

- **Excel reference for the lexicon lives at `/Users/robertpraas/Downloads/Cross-checked_AI terminology and taxonomy_analysis_final.xlsx`** (NOT in the project tree). It is the source of truth for terms, regulatory texts, article references, and terminology resolutions. See `iterations/v28_excel_inventory.md`.
- **Excel structure is sparse**: analysis sheets have header rows at row 1ÔÇô2, attribute labels in column A, and article references inline in data cells. Continuation rows (no label in col A) extend the cell above ÔÇö extraction must walk them, not assume one row per attribute.
- **Sheet name gotcha**: sheet 5 is `" High-risk AI system_ANALYSIS"` with a **leading space**. Always `sheet.strip()` when looking up by name.
- **Anchor-term pattern**: every U.S. state term has an EU-AIA "anchor" (e.g., Colorado "Developer" ÔåÆ "Provider of limited-risk AI systems"). The `Provider_Developer_Analysis` and `Deployer_Supplier_Analysis` sheets are 4 and 3 stacked sub-tables respectively, each defining one anchor ÔåÆ its U.S. equivalents. Without the anchor, "Developer"/"Deployer" appear ambiguously across multiple contexts.
- **CoP three-chapter routing**: Excel cites CoP CC (copyright), CoP TC (transparency), CoP SSC (safety/security) inline; v27 HTML splits these into three separate `law-blob`s (`eu-gpai-cop-copyright`, `eu-gpai-cop-transparency`, `eu-gpai-cop-safety`). When parsing inline citations, route by chapter prefix.
- **"12 regulatory frameworks" (per About sheet) vs Methodology's 13 rows**: the canonical count is 12 (Utah's SB 149 + SB 226 collapse to one Utah framework). v27 currently wires 15 blobs because it splits CoP into 3 and adds two extra Commission Guidelines (`eu-guidelines-ai-definition`, `eu-guidelines-prohibited`) that are NOT in Excel scope.
- **"Limited-risk" is not a defined AIA legal category** ÔÇö it's a Commission classificatory label tied to Article 50 + Recital 132. Surface this caveat in the UI rather than treating it as a statutory term.
- **Drawer verbatim is rendered via `textContent`, not `innerHTML`** (`drawer-verbatim` element, around line 1966 of the v28 HTML). HTML markup placed inside any `verbatim` field of a law-blob will appear literally to the user. For exponents and other typography that must render as glyphs in the verbatim drawer, use Unicode characters (e.g. `┬▓ÔüÁ`), not `<sup>` markup. Analysis cells and CEPS-notes themes/summary, by contrast, are rendered via `innerHTML` and accept HTML markup safely.
- **Exponent rendering pipeline (US-003)**: `build_v28.py` runs a final `apply_superscripts(html)` pass after all other swaps. Outside `<script>` blocks every form (`10^25`, `10**25`, `10(^25)`, Unicode `10┬▓ÔüÁ`) becomes `10<sup>25</sup>` markup. Inside `<script>` blocks (CONCEPTS literal + `<script type="application/json">` law-blobs) only ASCII forms are converted, and they go to Unicode superscripts so the `textContent`-rendered drawer still shows proper glyphs. The pass is idempotent: existing `<sup>` markup and existing Unicode superscripts are never re-wrapped.
- **Opening cluster matrix row labels live in EU variants, not `term_label` (US-004)**: `_renderClusterMatrix` in the HTML explicitly merges the term-label and EU columns ÔÇö its comment reads "We merge the two: render EU's variants as the term label stack". The visible row label IS the EU cell's `variants[].name`; the JSON's `term_label` is metadata only and never rendered. To rename an opening-table row, edit `cluster_summary.rows[N].cells.eu.variants[K].name`, not `term_label`. The U.S.-state cells in the same row are anchor *equivalents* (per Excel anchor-term pattern) and must NOT be renamed unless Excel says so.
- **Detail-table popups are scoped per cell, not per article reference (US-005)**: each table cell in a sub-concept's `dimensions[].cells[<jid>]` is a triple `{analysis, verbatim, reference}`. Clicking the cell triggers `openDrawer(dimId, juris)` ÔÇö the drawer reads `cell.verbatim` (rendered via `textContent` so HTML markup is forbidden ÔÇö use Unicode glyphs) and `cell.reference`. There's no inline-article-link mechanism: the cell IS the popup. To make a cell's popup link correctly, the verbatim and reference fields must agree with the article cited in the analysis text. Empty `verbatim` falls back to `"[Analysis text ÔÇö no verbatim extracted]\n\n" + analysis`, which is functionally a degraded popup.
- **Cell-triple anchors are non-unique across sub-concepts (US-005)**: in the limited-risk Provider/Developer table and the limited-risk Deployer/Supplier table, several cells share identical `{"analysis":...,"verbatim":"","reference":""}` triples (e.g., AI literacy EU, Transparency EU). When patching cells via `html.replace()`, scope the replacement to the specific sub-concept's JSON segment by slicing between sub-concept header markers (e.g., `"id":"provider","title":"Provider of limited-risk AI systems"` ÔåÆ `"id":"provider-of-high-risk-ai-systems"`). Don't rely on dimension ids (`transparency-4-0`) for global uniqueness ÔÇö they encode row position and recur across sub-concepts.
- **CO law-blob has no embedded sections (US-005)**: `<script id="law-blob-co-sb24205">` carries `"sections":[]` and a generic `raw_text` placeholder, so popups cannot pull authoritative Colorado verbatim from the existing data layer. For Colorado cells, set the `reference` field (label is shown in the drawer) but leave `verbatim` empty; the drawer falls back to analysis text + the "[Analysis text ÔÇö no verbatim extracted]" note. Don't fabricate verbatim quotes.
- **Lexicon regulatory scope is EU + US state only (US-007)**: the Excel's 12-text scope list (Methodology sheet) names only EU instruments (AI Act, CoP, GPAI Guidelines) and US state laws (CA SB 53/SB 942/AB 2013, CO SB 24-205/SB 25B-004, NY A6453B/S8828, TX HB 149, UT SB 149/SB 226). **No US federal text** (NIST AI RMF, OMB AI memos, FTC ┬º5, EO 14179, sector federal AI rules, etc.) is in scope. Audit/link-check stories that target "US federal" texts have an empty scope by construction.
- **CONCEPTS literal round-trips byte-for-byte via `json.dumps(..., ensure_ascii=False, separators=(',', ':'))` (US-008)**: the v26 build emits the CONCEPTS JSON in the exact form Python's `json.dumps` produces with these flags. Bulk per-cell fixes are far simpler this way than the per-cell `string.replace` anchors used in US-005/US-006.
- **State-law jids are `<state>` for shared content, `<state>-<idx>-<term>` for per-row variants (US-008)**: `jids` like `co`, `tx`, `ut`, `ny` are the bare state code; `ca-0-covered-provider`, `ca-1-developer`, `ny-0-frontier-developer`, `ny-1-large-frontier-developer`, `ny-2-large-developer`, `ca-0-substantially-modified-version-of-a-frontier-model-no-standalone-defined-term`, etc. are per-row variants when a state has multiple equivalent terms in one sub-table (per Excel `Provider_Developer_Analysis` ┬º4.4 etc.). Cross-state filters / audits must accept any jid that starts with `<state>` or `<state>-`.
- **NY references split between bills by section number (US-008)**: ┬º1427 and ┬º1428 only exist in S8828 (the amendment); ┬º1420ÔÇô┬º1426 in both S8828 and A6453. `ny-0-frontier-developer` and `ny-1-large-frontier-developer` carry S8828 content; `ny-2-large-developer` carries A6453 content (per Excel `Provider_Developer_Analysis` ┬º4.4).
